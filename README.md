## рџ§Є Testing Strategy

This project follows the **Testing Pyramid** approach with **Shift-Left** methodology:

```
           /\
          /  \    E2E вЂ” Full Stack
         /____\   Smoke tests on real env (P1 happy paths only)
        /      \
       /________\ E2E вЂ” FE + Mocked BE (Playwright)
      /          \ User flows вЂ” no real BE needed
     /____________\
    /              \ Integration + Contract (Pact)
   /________________\ API integration, consumer-driven contracts
  /                  \
 /____________________\
                        Unit (Vitest, JUnit5)
                        Business logic & component behavior
                        Target: >90% coverage
```

### Test Distribution
- **Unit Tests:** ~90% (business logic, component behavior)
- **Integration + Contract (Pact):** ~7-8% (API integration; Error states; Pact consumer/provider tests decouple FEв†”BE verification вЂ” no need to run both together)
- **E2E вЂ” FE + Mocked BE:** ~1-2% (browser-level user flows via Playwright `page.route()` вЂ” fast, no real BE required; confidence backed by Pact contracts)
- **E2E вЂ” Full Stack:** <1% (smoke tests on staging only вЂ” P1 happy paths to confirm deployment wiring, not to test business logic)

### Playwright Test Suites

The Playwright suite (`e2e/playwright-typescript`) is split into three distinct groups that map directly to the pyramid above:

| Suite | Tag | Purpose | When to run |
|---|---|---|---|
| Mocked BE | *(no tag)* | Browser-level user flows with `page.route()` mocks вЂ” no real BE needed | Every CI run |
| Accessibility | `@accessibility` | axe-core scans for WCAG violations on key UI states | Every CI run |
| UAT | `@uat` | Single smoke test against the real running app вЂ” verifies the critical create-task happy path end-to-end | Staging / post-deploy only |

The UAT suite is intentionally minimal (one test вЂ” the most critical happy path). Unit, integration, Pact, and mocked-BE E2E tests already cover business logic thoroughly; the UAT test exists purely to confirm that all services are correctly wired together in a real environment.

### Pact Consumer Contracts (Frontend)

Run Pact consumer tests for the frontend API client and publish to a standalone local Pact Broker:

```bash
# 1) Start standalone Pact Broker (Docker + Postgres)
cd demo-interface
npm run pact:broker:up

# 2) Generate contracts in demo-interface/pacts
npm run test:pact

# 3) Publish generated contracts to the broker
PACT_CONSUMER_VERSION=$(git rev-parse --short HEAD) npm run pact:publish

# 4) Optional deployment safety check
PACT_CONSUMER_VERSION=$(git rev-parse --short HEAD) npm run pact:can-i-deploy

# 5) Stop broker when done
npm run pact:broker:down
```

Default broker URL is `http://localhost:9292` and can be overridden with `PACT_BROKER_BASE_URL`.

### Pact Consumer Contracts (Notification Service)

Run Pact consumer tests for the `notification-service` Kafka listener and publish the generated message contract to the broker:

```bash
# 1) Start standalone Pact Broker (if not already running)
cd demo-interface
npm run pact:broker:up

# 2) Generate the message contract in notification-service/target/pacts
cd ../notification-service
mvn test

# 3) Publish the generated contract to the broker
docker run --rm --network=host \
  -v "$PWD/target/pacts:/pacts" \
  -e PACT_BROKER_BASE_URL=http://localhost:9292 \
  pactfoundation/pact-cli:latest \
  pact-broker publish /pacts \
  --consumer-app-version $(git rev-parse --short HEAD) \
  --branch $(git rev-parse --abbrev-ref HEAD)
```

The consumer test (`TaskEventPactConsumerTest`) defines that the notification service expects messages containing `taskId` (any string) and `status` (one of `TODO | IN_PROGRESS | DONE`). It invokes the real `TaskEventListener` against the Pact-generated message to verify the production deserialization path.

### Pact Provider Verification (Backend)

Verify that `demo-service` honours all consumer contracts published to the broker вЂ” both the HTTP API contracts from `demo-interface` and the Kafka message contract from `notification-service`.

All provider tests run under the same Maven command. Each test class fetches its own pacts from the broker and verifies them against the running service:

| Test class | Contract type | Consumer |
|---|---|---|
| `api/*PactProviderTest` | HTTP request/response | `demo-interface` |
| `event/TaskEventsPactProviderTest` | Kafka message | `notification-service` |

```bash
# 1) Start the Pact Broker (if not already running)
cd demo-interface && npm run pact:broker:up

# 2) Run all Pact provider verifications (integration-tests profile required)
cd ../demo-service
PACT_BROKER_BASE_URL=http://localhost:9292 mvn verify -Pintegration-tests -Dit.test="*PactProviderTest" -Djacoco.skip=true
```

**HTTP provider tests** start a full `@SpringBootTest(RANDOM_PORT)` with real Testcontainers (MongoDB, Kafka) and embedded WireMock, then replay each consumer interaction against the running service. Provider states set up the exact DB fixtures and WireMock stubs each interaction requires:

| Provider State | What it does |
|---|---|
| `task title is unique` | Ensures no task with the target title exists |
| `task title already exists` | Seeds a task with the target title |
| `tasks exist` | Seeds one task and returns its generated ID |
| `a task exists` | Seeds a task; returns `taskId` for path injection |
| `a task exists to delete` | Seeds a task targeted for deletion |
| `a task exists to update and title is unique` | Seeds one task; ensures the updated title is free |
| `another task has the requested title` | Seeds two tasks to trigger a duplicate-title conflict |
| `validation result is true for the task` | Seeds a task; stubs external validator to return `true` |

**Kafka message provider test** uses a `MessageTestTarget` вЂ” no HTTP server or broker required. The `@PactVerifyProvider` method serialises a real `TaskEvent` via the Spring `ObjectMapper`, and Pact verifies the resulting JSON satisfies the notification service's contract (`taskId` + `status` present and correctly typed).

> **Tip:** Publish both consumer contracts before running provider verification: frontend pacts via `npm run test:pact && npm run pact:publish`, notification-service pacts via `mvn test` + the publish docker command above.

### E2E Tests

Each E2E framework lives in its own sub-folder under `e2e/` and has its own section below.

#### Playwright + TypeScript

Tests run across Desktop Chrome and Mobile Safari (iPhone 12 Pro). Three suites can be run independently:

```bash
cd e2e/playwright-typescript
npm install

# Suite 1 вЂ” mocked BE + accessibility (no real backend needed)
npm run test:mock

# Suite 2 вЂ” accessibility only
npm test -- --grep @accessibility

# Suite 3 вЂ” UAT smoke test (requires the full app to be running)
npm run test:uat

# All suites
npm test
```

Copy `.env.e2e` to `.env.e2e.local` and set `E2E_TEST_ENV_URL` to point at the running frontend before executing UAT tests.

```bash
# View Allure report
npm run allure:generate && npm run allure:serve
```

See [Playwright E2E README](e2e/playwright-typescript/README.md) for full configuration options.

#### Selenide + JUnit5 + Selenium Grid (Java)

Tests cover both desktop and mobile viewports across five test suites: Create Task, Delete Task, Edit Task, Task Info, and Language Support.

**Option A вЂ” Local browser (no grid):**
```bash
cd e2e/selenide-junit5-selenium-grid
mvn -Dproperties.file.name=test.local.properties clean test
```
Set `test.env=local` in `src/test/resources/test.local.properties` before running.

**Option B вЂ” Selenium Grid in Docker:**
```bash
# One-time network setup
docker network create qa-demo-e2e

# Start application and Selenium Grid
docker compose -f docker/docker-compose/run-application.yml up -d
docker compose -f docker/docker-compose/run-selenium-grid.yml up -d

# Run tests
docker compose -f docker/docker-compose/run-tests-selenide-junit5-selenium-grid.yml up

# View Allure report (requires allure CLI)
rm -rf allure-results && docker cp qa-demo-selenide-junit5-selenium-grid-tests:target/allure-results allure-results && allure serve allure-results
```

See [Selenide E2E README](e2e/selenide-junit5-selenium-grid/README.md) for full configuration options.
